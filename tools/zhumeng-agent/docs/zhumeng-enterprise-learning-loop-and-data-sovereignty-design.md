## 0. 结论摘要与本文档的边界

### 0.1 一句话

逐梦在企业侧卖的不是"模型"，也不是"采集"，而是 **"让每个组织拥有、并能验证自己拥有那条把机构知识沉淀成复利资产的学习闭环"** —— 数据归客户、默认不出域、可审计、可换底层模型而不丢积累。逐梦只卖让这件事成立的基础设施（网关 + 治理 + 采集底座 + 闭环工具）。

这正是 Satya Nadella《A frontier without an ecosystem is not stable》(2026-06-14) 那句话的产品化：**"你可以外包一个任务甚至一份工作，但你永远无法外包学习本身。"** 在企业那里，"学习"就是机构知识的沉淀与复利；逐梦提供的是让它发生、且**安全地**发生的管道。

### 0.2 本文档与飞轮文档（文档1）的关系——那条"缝"

两份文档在两个高度，靠一条**归属/授权/分级策略**缝合：

```
文档2（本文）  企业定位 + 数据归谁 + 信任架构 + 商业模式      高度：产品/商业
     │  定义"归属/授权/分级/出境"策略（§2、§3）
     ▼
   〔缝〕 ownership + consent + tier + egress policy
     ▲
     │  实现并强制执行该策略（端侧分级、边缘脱敏、出境控制）
文档1（飞轮）  采什么 + 怎么采 + 各训练方法的数据出口        高度：数据工程
```

- **文档2 定义策略，文档1 实现策略。** 任何"哪一档默认采什么、什么默认不出域、出境前怎么脱敏"在本文定，文档1 照此落地（见 §8 对文档1 的最小改动清单）。
- 本文**不重复**文档1 已写好的采集字段与训练出口；需要采集机制时直接引用文档1。

### 0.3 本文档刻意不做的事（对齐 v3 §11.1 深度纪律）

v3 brief §11.1 明确："V1 ship 之前不写飞轮 RL 方案；每份文档的细节深度不得超过当前真实进度一个版本"，并点名 2300 行飞轮文档是"想得太远"。因此本文：

- **只写到定位 + 信任缝 + 近期可交付的闭环价值**，深度=领先现实一个版本；
- 所有训练方法级数据需求（RLVR、logprobs、loop 遥测、私有 eval 训练集等）**只在 §7 列名作为"未来留缝"，不展开**；
- 严谨分级（v3 §11.4）：100 分留给 §2 归属模型、§3 信任架构（碰 key/数据/合规）；60 分给 §4 闭环愿景、§7 远期清单。

### 0.4 v0.2 修订原则

本版采用一个产品/工程折中原则：

```text
能力预埋，默认分档，启用靠授权/合同/开关。
```

因此，逐梦要从早期就预埋数据积累、企业自有闭环、逐梦共建池、人员维度分析、skill/workflow 沉淀和训练出口；但默认姿态按客户档和数据档加严，企业业务数据默认不出域，企业贡献逐梦必须合同化 opt-in。

### 0.5 v0.3 修订原则

本版补三条边界：

1. **个人也有自有闭环**：个人 L2/L3 数据若开启，默认进入个人自有池（`owner=user`，local-first），贡献逐梦是独立 opt-in。
2. **人员维度默认聚合**：企业默认做团队/角色/项目/流程聚合分析；人级排名、失败率、token 异常、绩效相关分析必须由租户显式开启，并受员工告知、可见范围和当地合规约束。
3. **数据价值再利用与交易预埋**：客户留存数据需要数据资产工作台来清洗、标注、挖掘 workflow/skill、生成私有 eval/训练集；客户也可选择把 L0/L1/L2 或经审核 skill/workflow 打包贡献/交易，逐梦可作为收购方，但必须走授权、账本、对价和用途限制。

## 1. 为什么是现在：从 Nadella 洞察到逐梦企业定位

### 1.1 洞察

Nadella 的核心论点（发布后约 28–42M 阅读，亦有"自利解读"质疑，见 §1.3）：

- **风险**：若 AI 价值只沉淀到少数前沿模型里，模型会"吃掉它看到的一切"，把各行业的专长商品化、掏空护城河；"政治经济体不会容忍"。
- **解法**：建"前沿生态"而非只建"前沿模型"，让**每个组织拥有那条编码自身机构知识的学习闭环**，使**人力资本（Human Capital）**与 **Token 资本（Token Capital）**一起复利。
- **AI 主权**的真正考验 = **能换掉底层通用模型而不丢失积累的机构智能**。
- 配套批评 **token-maxing**：无脑用最贵的模型干所有事，不管便宜模型够不够用。

### 1.2 为什么这对逐梦是机会，而不是又一个远期愿景

把"采集"的主语从"逐梦拿数据训自己的模型"翻转成"**企业拥有自己的闭环**"，三件事同时成立：

1. **信任反转**：数据与产出归客户、默认不出域，逐梦只卖铲子——逐梦从"潜在的数据榨取方"变成"帮你建主权的供应商"。这正是质疑者眼中微软的位置（占 harness+context、乐见模型层商品化），但逐梦做的是微软在中国市场够不到的那块。
2. **与现有资产严丝合缝**：v3 判断 B 的"**网关为脊柱**"，本身就是 Nadella 说的"能换底层模型而不丢积累"的技术载体；v3 §11.3 要开源的 **gateway-lite**，正是让客户**敢信**"数据没回流逐梦"的硬解锁。
3. **填的是 v3 自己列的 V1 交付物**：v3 §3 把企业档定义为"掌控 + 合规 + 数据不出域 + **数据永不入池（并以此为卖点）**"，这里应理解为**默认不进入逐梦池/公共训练池**；若企业后续合同化共建，则另走逐梦共建池。§8 路线图把"企业私有化方案文档"列为 V1。本文是它的种子。

### 1.3 必须正视的质疑（对内自省）

对 Nadella 的主流质疑是"自利叙事"：他鼓吹模型层商品化，因为微软占着 harness 和 context、唯独不靠自有前沿模型。**逐梦若照搬，会被同样质疑。** 我们的诚实回应不是辩解，而是用架构兑现承诺：

- 凡是声称"不出域/归你"的，必须**可被客户独立验证**（§3 出境账本 + 开源 shim），而不是"相信我"；
- 逐梦自己练模型的诉求（飞轮文档主线）与企业主权**不混淆**：企业档默认业务数据零出境，逐梦的训练料只来自个人/团队显式 opt-in 或企业合同化贡献（§2、§5）。

## 2. 数据归属模型（全文地基）

### 2.1 两个不可混淆的维度：数据档 × 客户档

**数据档**（沿用飞轮文档 §3 定义，本文不重述字段）：

| 档 | 内容 | 含不含客户 know-how |
|---|---|---|
| L0 | 匿名产品指标（用量、成功率、延迟…） | 否 |
| L1 | 协议形状（schema、字段名、hash、错误类…），**不可还原内容** | 否 |
| L2 | **脱敏语义轨迹**（脱敏任务/输出/工具 IO、上下文摘要） | **是**（脱了 PII，没脱 know-how） |
| L3 / L3+ | 可训练原文 + Reasoning/CoT | 是 |

> **本文最重要的一条线：分级的判据是"含不含客户 know-how"，不是"企业觉得重不重要"。**
> L1 与 L2 隔着这条护城河——L1 是"系统/协议"知识，L2 是"业务"知识。一条脱敏后的"这家公司怎么排查支付故障"轨迹，PII 抹了，但**"怎么排查"这个判断力本身就是客户的护城河**。把 L2 当成"反正他们不在乎"而默认吸收，就是 Nadella 警告的那件事，且若客户的竞争对手也用逐梦，等于把 know-how 经由模型泄给对手（违反飞轮文档"禁止跨租户训练"）。

### 2.2 归属矩阵（默认行为）

原则：**内容类（L2/L3）一律默认关、opt-in；匿名 L0/L1 的默认随客户档加严。**

| 客户档 | 默认到逐梦 | 可选（合同/对价 opt-in） | 出境账本须证明 |
|---|---|---|---|
| **个人 / 团队** | 匿名 **L0/L1**（指标 + 协议形状，全 hash，不可还原内容）；个人/团队自有 L2/L3 默认不进逐梦 | 显式加入"共建计划"贡献 **L2/L3** 或数据包，换折扣/收益/共建权益 | "只有这些 hash/指标/已授权数据包出去了" |
| **企业 / 私有化** | **业务数据零出境**；控制面元数据最小化、可配置、全记账 | 合同明写可选贡献（如跨组织 benchmark、共建模型、行业 skill），**单独谈对价** | 业务数据零出境；控制面与贡献项逐条可验 |

说明：

- 个人/团队默认 L0/L1，对齐 v3 决议5"默认只采 L0/L1（指标+协议形状），不采任何原文"——逐梦要靠它跑漏斗、量 v3 §8 留存数字、优化网关/适配器。若个人/团队显式开启 L2/L3，默认进入个人/团队自有池，用于自己的工作流、skill、私有 eval 或个人可携带 AI 工作记忆，不默认贡献逐梦。
- 企业档“业务数据零出境”是 v3 §3“数据永不入池”卖点的兑现；授权心跳、license、billing、模型目录、安全公告等控制面元数据可按合同最小化出境，但必须可配置、可审计、写入出境账本。想要增值（如行业 benchmark）走 opt-in 对价，**默认关 + 可加** 比 **默认开 + 可减** 更灵活也更可信。
- **L2/L3 在任何档都默认关。** 这是 option-3"默认关 + opt-in"原则的全局落点。

### 2.3 归属随数据走

每条采集事件出生即带不可篡改的归属标签（具体字段在文档1 实现）：

- `owner`：tenant / user / 逐梦（默认 tenant 或 user，绝不默认逐梦）
- `tier`：L0/L1/L2/L3/L3+
- `egress_allowed`：该条是否允许离开客户边界（企业档默认 false）
- `contribution_consent_id`：若 opt-in 反哺逐梦，引用对价授权
- `tenant_id`：租户隔离与删除的锚点

### 2.4 三个数据池

为避免“客户主权”与“逐梦模型反哺”混用，所有数据必须落入三池之一：

1. **客户自有池**：企业/团队/个人自己的 AI 工作资产池，用于工作流盘点、私有 eval、私有训练、组织/个人 skill、员工/个人偏好和治理审计。个人用户开启 L2/L3 后默认进入个人自有池（`owner=user`，local-first）；企业档默认只进入租户自有池。
2. **逐梦共建池**：用户/租户显式 opt-in 或企业合同化贡献后，进入逐梦可使用的数据池。该池可用于产品改进、共建模型、行业 benchmark、共建 skill。
3. **公共训练池**：从逐梦共建池中再经过授权、脱敏/扫描、质量、license、teacher_eligible 检查后，才可进入逐梦模型训练。

逐梦模型训练反哺不是当前企业默认交付，但共建池和公共训练池的授权口、标签和导出能力必须预埋。

## 3. 信任架构：那条"缝"（文档1 要实现）

"采集能否成为竞争力"的前提是：架构让客户**一眼看清、且能独立验证**数据在他这边、模型商与逐梦都看不到。否则"采集"只会吓跑客户。这一节是全文的 100 分区。

### 3.1 端侧分级（fail-closed）

- 每条采集事件在**本地 shim** 完成 tier 判定后，才决定能否出境。
- **判据是"含不含 know-how"**（§2.1）；判定器拿不准时**往高判**（默认当 L2 处理、不出境），即 fail-closed。
- 这要修正飞轮文档现状：飞轮文档 §7.0/§8.3 把"原文优先在服务端采、服务端再脱敏"——意味着原文已越过客户边界才脱敏，与主权承诺打架。**正确方向：分级与脱敏在边缘发生，L2/L3 原文在本地处理完之前一个字节都不出境**（见 §8 改动清单）。

### 3.2 边缘脱敏（出境前）

- 任何被允许出境的数据，脱敏在**出境前、本地**完成（密钥/token/PII + PIPL 下的第三方个人信息）。
- 服务端只做**二次校验**（defense in depth），不作为首道防线。

### 3.3 业务数据/控制面出境边界与出境账本

“零出境”必须拆开说，避免企业审计时变成无法兑现的绝对承诺：

- **业务数据**：任务、prompt、response、工具 IO、workflow、skill、文档、代码、Reasoning/CoT 等默认零出境。
- **控制面元数据**：license、授权心跳、billing、版本检查、安全公告、模型目录等可按合同最小化出境。
- **贡献数据**：企业明确共享给逐梦的 benchmark、workflow、skill、训练样本等，必须合同化 opt-in，并进入逐梦共建池。

信任不能靠“相信我隐私模式有效”。必须给客户一份**客户自持、防篡改、可逐条核对**的出境清单：

- 记录到底什么离开了客户边界；
- 控制面元数据也记账；
- 贡献给逐梦的数据也记账并绑定 `contribution_consent_id`；
- 企业业务数据零出境时，账本应能证明无 L2/L3/L3+ 业务数据出境；
- 服务端接收回执也回写账本，形成客户侧可验闭环；
- 具体 hash 链、签名、Merkle 等构造留 V1 设计，本阶段只定原则。

### 3.4 租户隔离与删除/撤回

沿用飞轮文档 §11.3/§11.4：tenant 级 opt-out、tenant 级 trainable=false、私有微调池、禁止跨租户训练、撤回与历史删除。本文把它们从"合规约束"提升为**产品头条卖点**。

### 3.5 开源 gateway-lite = 信任硬解锁（对齐 v3 §11.3）

- 经手 key/流量、做端侧分级与出境控制的那层 shim **应尽早开源**，让客户可审计"碰 key、决定出境"的逻辑；
- **协议适配的深度逻辑仍闭源**（v3 §6 护城河第 1 条）；
- 开源范围 = "碰 key 与决定出境的壳"，不是智能路由，也不是适配深度。

## 4. 企业学习闭环（定位深度，不展开训练细节）

### 4.1 近期王牌：不训模型也能用的"工作流盘点/优化面"

绝大多数企业短期**不会**训自己的基座模型。逼它们训模型是错的预期。**就算一行模型都不训**，沉淀的数据也能直接形成闭环价值，这应是企业档的**楔子/MVP**：

- 回答 CEO 真正关心的问题：我的人到底在拿 AI 干什么？哪些任务该 agent 化？哪里反复失败？哪些 prompt/skill/MCP 该标准化成公司资产？哪里在 token-maxing（贵模型干了便宜模型能干的活）？
- 它就是 Nadella 的 learning loop，但落在**流程/harness 层**而非模型层——而流程层正是小团队能交付、微软在中国够不到的层。
- 交付物形态：给企业管理员看的盘点/优化看板 + 可沉淀的"公司工作流/skill 资产库"。**不碰训练**。

### 4.2 这套闭环要求采集的数据（飞轮文档当前缺，按定位深度列，不出 schema）

为支撑 §4.1，文档1 需补以下采集对象（**为什么要采**，字段留给文档1）：

1. **组织/角色/团队图谱**：盘点的前提。飞轮文档只有 user/tenant 的 hash，缺 role/team/部门维度。
2. **工作流作为一等对象**：识别"N 个员工在做同一件多步任务"→ 聚类 → 沉淀黄金轨迹/公司 skill。飞轮文档有 episode/turn/step，但没有"重复工作流"这个概念。
3. **成本/ROI/token-maxing 归因**：用贵模型 vs 便宜模型结果有没有区别、省了多少人工。这是把"飞轮"翻译成决策层语言、也是 token-maxing 叙事的数据。
4. **多人交接**：企业工作流常跨人（起草→审→批），飞轮文档 §5.13 的多 agent 全是 AI 子 agent，缺人-人交接。
5. **企业 system-of-record 数据血缘**：从哪个系统读、写到哪个系统（Jira/Wiki/CRM），是机构知识的流动路径。
6. **自然标注流（推广到所有任务类型）**：资深员工修正初级员工的 AI 产物 = 天然金标签；飞轮文档只在编程场景有 `post_agent_human_fix_diff`。

> 这些都是企业**自有闭环**所需，默认归客户；逐梦默认不参与。若企业合同化共享，可按分级菜单贡献 L0/L1 聚合、脱敏 L2、指定 workflow/skill、行业 benchmark 或行业模型共建数据，且必须进入逐梦共建池。

### 4.3 人员维度 AI 采用度与能力分析

企业通常会需要看到 AI 采用度：哪些团队/岗位/流程 AI 用得好，哪里失败率高，哪里 token 消耗异常，哪些岗位需要培训。这是企业治理的自然诉求，产品能力应具备，并限定在企业租户、企业工作区和企业授权工具链内。

默认姿态必须是**团队/角色/项目/工作流聚合分析**，而不是默认做人级绩效画像：

- 默认展示组织、团队、角色、项目、工作流层级的聚合指标；
- 人级排名、个人失败率、个人 token 异常、个人 skill 贡献等能力预埋，但作为独立模式关闭；
- 人级分析需租户管理员显式开启，并配置员工告知、用途边界、可见范围、导出权限和保留期限；
- 是否用于绩效、是否员工可见、是否需要单独同意，由企业合同、组织政策和当地合规要求决定；
- 产品文案应定位为“AI 采用度与能力分析”，避免默认表达成监控员工。

一句话：**能力默认具备，默认计算和展示聚合；人级分析显式开启。**

### 4.4 Skill / Workflow 资产生命周期

企业里很多 know-how 来自“老师傅经验”：个人长期积累的判断、步骤、检查清单和处理习惯。逐梦的闭环要支持把这些经验从个人脑中沉淀为组织资产。

生命周期：

```text
员工真实使用轨迹
  -> 高频任务/高成功率轨迹聚类
  -> 黄金轨迹提取
  -> 生成候选 workflow / skill
  -> 员工或专家审核
  -> 发布到企业 skill 库
  -> 组织复用
  -> 持续采集使用效果
  -> 版本迭代
```

归属与上传机制：

- 候选 skill/workflow 默认归员工、团队或租户，按企业策略决定；
- 员工可自主提交最终成品 skill 到企业 skill 库；
- 企业可配置审核流，把个人经验升级为组织资产；
- skill 内容本身是否贡献给逐梦，必须单独 opt-in；
- 但 skill 使用过程中的基础指标、成功率、调用链路和版本效果，应按企业策略进入客户自有池，用于持续改进。

### 4.5 "能换模型不丢积累" = 网关的副产品

Nadella 的"AI 主权考验"——换底层模型不丢机构智能——在逐梦这里**几乎是网关的免费副产品**：闭环资产（评测集、工作流库、skill、路由偏好）沉淀在客户侧、与具体模型解耦，底层模型经网关随时可换。这点要在企业叙事里讲透，因为它是逐梦独有的结构优势。

### 4.6 数据价值再利用：客户数据资产工作台

客户留存“所有数据”之后，真正需要的是一套把日志变成资产的机制。逐梦应把数据资产工作台设计为企业/个人共用底座：

```text
数据盘点 -> 分级/权属/授权校验 -> 脱敏/去重/去污染 -> episode 修复 -> 质量评分/自然标注 -> workflow/skill 挖掘 -> 私有 eval/训练集/看板/导出
```

企业侧输出：

- 工作流盘点与优化建议；
- 高频任务、失败集中点、token-maxing、模型路由建议；
- 公司 workflow / skill 资产库；
- 私有 eval、私有微调/强化训练候选集；
- ROI dashboard 与治理审计报告。

个人侧输出：

- 个人工作流复盘；
- 个人 prompt/skill/偏好资产；
- 个人可携带 AI 工作记忆；
- 个人 eval 或个人模型偏好数据；
- 可选择贡献/交易的数据包预览。

### 4.7 数据贡献与交易：客户可控的数据包机制

对于不打算自行训练模型、但希望变现数据的个人/团队/企业，逐梦应提供“数据包贡献/交易”机制，而不是隐式吸收。

流程：

```text
客户选择数据范围和档级
  -> 生成可预览 dataset_package
  -> 做 PII/secret/IP/商业秘密/第三方个人信息检查
  -> 给出质量评分和估值建议
  -> 客户确认用途、期限、对价和撤回规则
  -> 签署 contribution_consent / data_purchase_agreement
  -> 写入出境账本和交易账本
  -> 进入逐梦共建池
  -> 按授权再进入行业 benchmark 或公共训练池
```

可交易数据形态：

- L0/L1：聚合指标、协议形状、工具错误类型、模型路由统计；
- L2：强脱敏 workflow 样本、行业 benchmark、经审核 skill/workflow；
- L3/L3+：默认不交易，只在特定共建模型/私有训练合同下开启；
- 逐梦可作为收购方购买授权数据，也可以未来作为撮合方，但必须保证客户可预览、可定价、可限定用途、可追踪账本。

这套机制的产品价值是让客户既有掌控感，又能看到数据的经济价值；逐梦获得的是合规、可证明来源的数据供给，而不是灰色日志池。

## 5. 商业模式：数据红利对价阶梯

把 §2 的"两条腿"做成**销售条款**而非默认行为，对齐 v3 §5 定价：

| 客户档 | 默认数据姿态 | 对价阶梯（可选 opt-in） |
|---|---|---|
| 个人（Free/Token） | L0/L1 默认；L2/L3 显式开启后默认进个人自有池 | 加入"共建计划"或出售数据包 → 额度/权益/收益 |
| 团队（座位订阅） | L0/L1 默认；团队 L2/L3 默认进团队自有池 | opt-in 贡献 L2/L3 或 workflow/skill 包 → 折扣 / 共建模型权益 / 收益 |
| 企业（License/私有化） | **业务数据零出境；控制面最小记账；自有数据资产工作台** | 合同对价：跨组织 benchmark、共建模型、行业基线、行业 skill、数据包交易 → 商务定价 |

原则：

- **对价必须是客户清楚知道自己在交换什么的显式交易**，绝不在 L1 之上做静默默认吸收（一旦被审计/泄露捅出，"卖铲子"信任当场崩盘——对小公司是致命级声誉事件）。
- L1 反哺也要**聚合/hash、跨多租户汇总后**才进任何模型，不留单租户可识别痕迹。
- 个人/团队/企业愿意共享或交易时，优先设计为分级菜单：只共享 L0/L1、共享脱敏 L2、共享特定 workflow/skill、共享行业 benchmark、共建行业模型或出售数据包。
- 与 v3 判断 D 对齐：durable 收入在**团队/企业治理**，数据对价是治理价值的延伸，不是替代。

## 10. 一句话总结

> 逐梦企业/个人侧 = **帮每个组织和个人拥有、并能验证拥有自己的学习闭环**：数据归客户，个人默认进个人自有池，企业业务数据默认不出域，控制面最小可审计，可换模型而不丢积累。逐梦只卖让这件事成立且安全的铲子；客户若愿意共享或交易，则通过合同化共建进入逐梦共建池。**先把信任缝（§3）做成可验证的，再谈闭环的性感（§4）；一切节奏服从 v3。**
