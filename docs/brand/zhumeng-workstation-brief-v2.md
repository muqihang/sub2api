## 2. 产品形态:逐梦工作台

### 2.1 它是什么

一个统一的本地桌面 App:

- 中文:**逐梦工作台**
- 英文:**Zhumeng Workstation**

用户下载一次、安装一次,获得三件套:

- 一组已魔改适配过的顶级 AI 工具(Augment、Codex、Claude Code、DeepRotor 等)
- 一个本地运行的模型网关(可以接入各家供应商 key)
- 一套可选的云端服务(SLA、补位线路、托管路由)

### 2.2 三模块架构

#### 模块 A:Tools(工具中心)
- 管理所有 Agent 注入工具的安装、启动、更新
- 当前覆盖:逐梦 Augment Code、逐梦 Codex、逐梦 Claude Code、DeepRotor
- 工具的魔改配置由工作台自动注入,用户零配置
- 未来可扩展第三方工具接入(Plugin 机制)

#### 模块 B:Gateway(本地网关)
- 基于现 sub2api 网关打包,作为本地服务运行(默认 localhost)
- 支持 BYOK:任意数量供应商
- **智能路由**(核心差异化):按任务类型自动选择最优模型组合
- 统一暴露 OpenAI 兼容接口
- 本地用量看板

#### 模块 C:Cloud Bridge(云端桥接,可选,境外)
- 默认**关闭**,主动启用时弹窗 + TOS 确认 + 本地日志
- 仅向境外用户提供
- 适用场景:用户无 key、网络差、希望托管路由

### 2.3 一个安装包按地区切 UI

**一个安装包**,首次启动时根据 IP / 用户主动选择决定显示哪种模型清单:

- 国内 IP → 默认显示**合规模型清单** + 逐梦合规云(中后期)
- 境外 IP → 默认显示**全模型清单** + Cloud Bridge
- 用户可主动切换,但需要协议确认 + IP 日志记录

理由:两个安装包维护成本高,且容易被认定为"针对国内做特别版"。一个 App 按上下文切换 UI,姿态更像"全球软件,地区适配"。

### 2.4 产品口号

**主口号:**
> **"你的桌面 AI 工作站。所有顶级工具,本地运行,统一调度。"**

**副口号:**
- BYOK 角度:**"你的 key,你的钱,你的工具。"**
- 工具角度:**"装一次,所有顶级 AI 工具开箱即用。"**
- 企业角度:**"私有化的 AI 工作台,数据不出本地。"**
- 路由角度:**"GPT、Claude、DeepSeek,按任务智能混调。"**(仅境外版)

## 3. 定价(v2 修订)

### 3.1 设计原则

1. **Free 必须真正可用** — 对抗 Trae / 通义灵码 / 文心快码 等国内大厂免费策略
2. **Pro 价值锁在"软件能力"** — 智能路由、配置同步、深度分析
3. **个人版是漏斗,团队版是金矿**
4. **定价远低于国际同行**(Cursor $20 / Tabnine $12 / Codeium $15),但高于"免费"

## 4. 合规姿态:国内/境外双线分割

### 4.1 核心设计

把业务拆成两条物理和品牌都分割的线:

| 维度 | 国内线(桌面工作台) | 境外线(Cloud Bridge) |
|---|---|---|
| 业务范围 | 桌面软件 + 私有化 + (中后期) 合规云 | 国内全部 + Cloud Bridge 未备案中转 |
| 法律定性 | 软件供应商 + 互联网信息服务 | 跨境互联网服务 |
| 目标用户 | 国内 + 海外个人/团队 | 仅境外用户 |
| 数据流向 | 本地 + (中后期) 合规云 | 经境外服务器 |
| 货币 | 人民币 | USD |
| 域名 | .cn + ICP 备案 | 境外域名 + 境外主体 |
| 主体 | 中国实体 | 新加坡/香港/美国实体 |
| 品牌 | 逐梦 / Zhumeng | **独立品牌**(待定) |

### 4.2 做实双线分割的 8 个关键设计

1. **桌面 App 内 Cloud Bridge 默认关闭**,主动启用时弹窗 + TOS 确认 + 本地日志
2. **国内流量入口绝不主推** Cloud Bridge,只讲软件 + BYOK
3. **货币和支付通道物理隔离** — 人民币不能直接买 Cloud Bridge token
4. **教育/OSS 优惠仅限桌面 Pro**,不送 Cloud Bridge token
5. **Cloud Bridge 端地理屏蔽做实** — IP 库 + 国别注册 + 海外支付验证 + 异常拦截
6. **现 sub2api 后端如部署在境内必须迁到境外**(新加坡/香港/日本节点)
7. **双品牌、双域名、双主体**,刻意做得不像姐妹品
8. **国内员工统一话术**:"Cloud Bridge 是另一个境外品牌的服务,不向大陆用户推荐"

### 4.3 国内合规云的"模型白名单"

国内合规云端**只接已通过深度合成算法备案的大模型**:

- DeepSeek(V3 / R1)
- 通义千问 Qwen(2.5 / Coder / VL)
- 文心一言(4.5 Turbo)
- Kimi(Moonshot)
- 智谱 GLM-4 / CodeGeeX
- 腾讯混元
- 字节豆包

**白名单写到代码里**。国内云端的路由器不允许出现 GPT/Claude/Gemini 等未备案模型。

### 4.4 最大风险:自己人打通两条线

典型出事场景:
- 国内运营为冲 KPI 在国内社区推 Cloud Bridge
- 桌面 App 默认配置里写了 Cloud Bridge endpoint
- 国内客服工单告知用法
- 国内发票出现境外条目

规避方法:两条线的**运营、技术、客服 人员、邮箱、Slack 频道物理隔离**。

## 5. 国内业务三阶段路径

| 阶段 | 时间 | 国内站做什么 | 需要资质 |
|---|---|---|---|
| **Phase 1** | 现在 - 6 个月 | 桌面 App 下载 + 私有化方案展示,只卖软件 | 公司主体 + 软著(可选) |
| **Phase 2** | 6-12 个月 | 增加官网/文档/控制台,网站正式上线 | **ICP 备案**(标准网站类) |
| **Phase 3** | 12-24 个月 | 提供国内合规云端服务 | **生成式 AI 服务备案**(深度合成算法备案) |

### 5.1 Phase 1(当下)

- **不需要 ICP 备案** — 桌面 App 下载可以临时跑在境外或做静态发布
- 不提供任何"互联网信息服务",只是软件下载
- 重点:把桌面 App 稳定打包,跑通人民币订阅支付
- 团队/企业销售可以开始(私有化方案推销)

### 5.2 Phase 2(有了第一批用户后)

- 申请标准 ICP 备案(公司主体名义)
- 上线官方网站、文档中心、用户控制台
- 用户可以在控制台管理订阅、查看本地用量(上传)、申请发票
- **此阶段仍不提供云端模型 API**

### 5.3 Phase 3(业务稳定 + 备案准备就绪)

- 申请生成式 AI 服务备案(深度合成算法备案)
- 接入已备案大模型,启动**逐梦合规云**
- 桌面 App 国内版默认显示合规云为云端选项
- 团队 / 企业版 SaaS 可基于合规云提供

**Phase 3 是难关:** 备案需要先有产品、有用户,提交算法描述、训练数据情况、安全评估,走属地网信办流程。好消息是我们**不训练模型,只调用已备案大模型**,路径相对简单。

## 6. 境外业务定位与品牌隔离

### 6.1 境外业务做什么

- 桌面工作台(跟国内同一个 App,只是 UI 显示全模型清单)
- **Cloud Bridge:** API 中转服务,核心业务
- 团队版 / 企业版 SaaS(全模型版本)
- 开发者 API 接入(OpenAI 兼容)

### 6.2 境外品牌建议

**桌面工作台:** 保留"逐梦 / Zhumeng"品牌(因为是同一个 App)

**Cloud Bridge / API 中转:** **独立品牌**,跟"逐梦"主品牌脱钩。备选:

- **RoutePort**(路由港,贴近 OpenRouter 命名风格)
- **MeshGate**
- **Cogni-Routes**

理由:监管追查时,看到的是"逐梦做软件,RoutePort 做 API 中转"两个独立服务,而不是"一家公司的两条业务线"。

### 6.3 境外流量获取

**只在境外渠道获取流量**:

- GitHub(开源项目、文档、Star)
- Hacker News / Reddit(技术社区)
- X / Twitter(开发者圈)
- 海外开发者论坛(dev.to, Indie Hackers)
- 英文 SEO(关键词:AI gateway, LLM routing, BYOK)

**国内 SEO / SEM / 信息流广告 / 公众号软文 → 绝不出现 Cloud Bridge 或境外品牌名。**

### 6.4 境外业务的目标用户

- 海外华人开发者
- 境外华人企业
- 真正的英语开发者(长期目标)
- "翻墙过来的国内用户" **不是目标用户** — 看不见、不主动服务,但允许存在

## 10. 开源战略

### 10.1 设计原则

走 **Open Core + Open SDK** 混合路线,分层处理:

- 能让"用户信任 BYOK + 本地运行"的部分**开源** — 信任的硬证据,合规姿态的加分项
- 形成体验护城河 + 商业核心价值的部分**闭源**
- **真正的工程护城河延迟开源** — 先建立技术领先,再用 BSL 释出

判断标准一句话:

> **"能让用户信任 BYOK 的部分开源,能形成体验护城河 + 商业核心价值的部分闭源。"**

---

### 10.2 协议适配层是 Zhumeng 的核心工程护城河

> 本节单独成节,因为这是 v2 战略下整个团队需要对齐的最重要判断之一。

**协议适配层 (Protocol Adapter)** 的定义:让一个模型"假装是另一个模型"被工具 App 原生调用 —— 比如让 DeepSeek 通过 Responses 协议 + app.json v2 在 Codex App 里被原生调用,99% 接近 GPT 的丝滑体验。

**这层包含的工程工作:**

- 逆向 Codex / Augment / Claude Code 的特定协议(Responses, app.json v2 等)
- 流式 SSE 的 delta 顺序对齐(OpenAI vs Claude vs DeepSeek 都不一样)
- tool_use / function_call 格式翻译
- system prompt 差异处理(OpenAI system role vs Anthropic system param vs DeepSeek 方式)
- thinking / reasoning 字段适配(O1 / DeepSeek-R1 thinking vs Claude extended thinking)
- vision content 格式差异
- 超长上下文截断逻辑
- 错误码统一映射
- Rate limit backoff 策略
- Prompt caching 兼容(Claude cache control vs OpenAI 差异)

**关键判断:**

1. **这是 Zhumeng 最核心的工程资产,不是"智能路由"。**智能路由是上层规则引擎(if-else + 策略调优),工程含量比协议适配低很多。
2. **协议适配赛道有"赢家通吃"特征。**用户只会信任做得最好的那家(踩坑成本太高),一旦某家把覆盖面、稳定性、边界 case 做到 99%,其他家很难追。
3. **LiteLLM / OpenRouter 没有做这一层**,它们只到"OpenAI 兼容接口"层就停了,没有针对具体工具 App 做深度适配。
4. **绝对不能在这个阶段开源协议适配层** — 等于把建立护城河的机会让给竞品。

**当前 6 个月内的战略要求:**

- 集中工程精力,把协议适配的覆盖面 + 丝滑度做到行业第一
- 覆盖更多模型(Claude 系列、DeepSeek 系列、Qwen 系列、Kimi、智谱、字节豆包...)
- 覆盖更多工具 App(Codex、Augment、Claude Code、Cursor、Windsurf、Continue...)
- 把"在 Codex 里跑 DeepSeek 像原生 GPT 一样"做成大家口口相传的标志性体验

#### Tier 1:完全开源(MIT / Apache 2.0)

### 10.5 运营成本与第一步建议

**开源不是免费午餐**,需要:

- 兼职 DevRel:0.5-1 FTE(文档、社区、PR review)
- Issue / PR 响应 SLA:48h 内回复
- 文档维护:README、QuickStart、API ref
- 社区建设:GitHub Discussions / Discord / 微信群
- 季度发版 + Changelog + 升级指南

**强烈推荐的第一步:先把 `workstation-tools` 仓库开源。**

理由:

1. **成本最低** — 本来就是脚本和配置,没有商业秘密
2. **回报最高** — 社区添加新工具一个 PR 就行,增长最快
3. **品牌建立** — Star 数最容易上来,GitHub 搜索权重最容易拿
4. **合规加分** — 国内监管看到"这家公司开源了工具注入逻辑",姿态更干净
5. **不损害商业护城河** — 真正的护城河是**协议适配层 + 桌面 App**,这两者依然闭源

这一步几乎零成本马上能做。其他层开源可以等到 Workstation v1 发布前再走。

> **v2 在第 10 章额外明确:协议适配层是 Zhumeng 真正的工程护城河,这是整个团队需要对齐的判断 — 不是"智能路由",不是"工具注入",是协议适配层。这一层 1-2 年内绝对闭源。**
