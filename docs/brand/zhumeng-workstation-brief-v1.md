# 逐梦工作台 (Zhumeng Workstation) 产品定位 Brief v1

> 本文档是在 [zhumeng-intent-brief.md](./zhumeng-intent-brief.md) 之后的**战略迭代**:把"卖工具 + 卖 Token"进一步收敛为"卖一个统一的本地 AI 工作台,云端服务为可选补位"。
>
> 后续品牌设计、主页改版、工具落地页规划、文档入口规划应以本文为新基准。

---

## 0. 关键升级一句话

**从** "我们是一个提供高价值 AI 工具的中转平台(卖电器)"

**变成** "我们是 AI 时代的本地工作台软件商,所有 AI 工具和模型路由能力都装在你自己的电脑/服务器里"

电(Token 中转)从"主营业务"降级为"可选的云端补位服务"。这一步看似只是产品形态变化,实质上把公司的法律定性、商业模式、护城河、用户心智**全部重构了一遍**。

---

## 1. 当前困境与战略再升级

### 1.1 v1 战略(已写在 zhumeng-intent-brief 里):从"卖电"到"卖电器"

原版思路是用工具把 Token 的差价"包装"起来,跳出价格内卷。但仔细推演,还有两个未解的问题:

1. **Token 利润长期趋零是必然的。** OpenRouter、各家直采、官方调价都在持续压缩中间商利润。把工具做漂亮只是把这个问题往后推一段时间,而不是解决它。
2. **法律定性没变。** 哪怕我们说"卖电器",底层依然是 API 中转,境内合规风险一直悬在头上,大客户、教育/政企的接入空间被压得很窄。

### 1.2 v2 战略:从"卖电器"到"卖整个工作台"

把整个 sub2api 网关 + 所有 Agent 注入工具,**打包成一个用户本地运行的桌面 App**。

- 用户买的是"软件",不是 API 服务 → 法律定性从"API 中转"变成"软件供应商"
- 用户用自己的 key (BYOK) → 数据不经过平台,合规风险大幅降低
- 软件订阅是稳定毛利,Token 差价是流量增量收入,Enterprise 是大单
- 平台占住"用户工作流的路由层"位置,粘性远高于 Token 销售

电(云端 Token)依然是收入来源,但**变成可选,不再是命脉**。

## 2. 产品形态:逐梦工作台

### 2.1 它是什么

一个统一的本地桌面 App,暂定名:

- 中文:**逐梦工作台**
- 英文:**Zhumeng Workstation**

用户下载一次、安装一次,获得三件套:

- 一组已魔改适配过的顶级 AI 工具(Augment、Codex、Claude Code、DeepRotor 等)
- 一个本地运行的模型网关(可以接入各家供应商 key,智能路由)
- 一套可选的云端服务(SLA、补位线路、托管路由 = 现有 sub2api 云端)

### 2.2 三模块架构

整个 App 内部分三大模块:

#### 模块 A:Tools(工具中心)

- 管理所有 Agent 注入工具的安装、启动、更新
- 当前覆盖:逐梦 Augment Code、逐梦 Codex、逐梦 Claude Code 桌面、DeepRotor 学术版
- 工具的"魔改"配置(模型注入、环境优化、端口/代理设定等)由工作台自动注入,用户零配置
- 未来可扩展第三方工具的接入(Plugin 机制),做成"AI 工具应用商店"

#### 模块 B:Gateway(本地网关)

- 基于现有的 sub2api 网关打包,作为本地服务运行(默认监听 localhost)
- 支持 BYOK:用户自行添加 OpenAI / Anthropic / DeepSeek / Moonshot / 自建 endpoint 等
- **智能路由**(核心差异化):按任务类型(代码补全、长上下文、视觉、推理)自动选择最优模型组合
  - 例:Codex 中,代码骨架走 DeepSeek,长上下文走 Claude,视觉理解走 GPT-4o
- 统一暴露 OpenAI 兼容接口,第三方应用(IDE 插件、脚本)可以直接接进来
- 本地用量看板:各供应商、各模型的调用次数、token 数、成本估算

#### 模块 C:Cloud Bridge(云端桥接,可选)

- 用户可以在 Gateway 配置中勾选"接入逐梦云"
- 启用后,网关会把指定路由(或全部路由)发到逐梦云端,继续走平台 token
- 适用场景:
  - 用户没有某家供应商 key,但需要用该模型(例如临时调用 Claude)
  - 用户网络条件不稳定,云端线路质量更好
  - 用户希望省去 key 管理麻烦,直接用平台一站式
- 这是 Token 销售在新模式下的延续:**不再强制,但永远可选**

### 2.3 产品口号

主口号:

> **"你的桌面 AI 工作站。所有顶级工具,本地运行,统一调度。"**

副口号(可在不同场景轮用):

- BYOK 角度:**"你的 key,你的钱,你的工具。"**
- 工具角度:**"装一次,所有顶级 AI 工具开箱即用。"**
- 企业角度:**"私有化的 AI 工作台,数据不出本地。"**
- 路由角度:**"GPT、Claude、DeepSeek,按任务智能混调。"**

## 3. 用户分层与商业模式

### 3.1 三条腿走路

**① 软件订阅(基本盘)**
- 用户为"工作台软件"付月费/年费
- 不依赖 Token 调用量,稳定可预测
- 对标:Cursor (¥20/月起)、Raycast Pro (¥10/月起)、Linear (¥10/月起)

**② Token 流量(增量)**
- 通过 Cloud Bridge,继续承接没有 key 或网络差的用户调用
- 这部分相当于把现有 sub2api 业务平滑迁过来,不损失存量
- 利润空间不大,但承担"漏斗补完"和"用户初始留存"的作用

**③ Enterprise 大单(高客单价)**
- 私有化部署(在客户机房/内网跑一份工作台 + 网关)
- License + SLA + 定制开发服务
- 高校实验室、企业研发团队、政企客户是核心目标

### 3.2 定价档位(待最终拍板)

| 档位 | 目标用户 | 包含内容 | 价格 |
|---|---|---|---|
| **Workstation Free** | 个人尝鲜 | 工具中心 + 本地网关(限 1 个供应商,基础路由) | 免费 |
| **Workstation Pro** | 个人重度用户 | 多供应商混调 + 智能路由 + 用量看板 + 全部工具 | ¥69-99/月 |
| **Workstation Team** | 小团队(5-20 人) | Pro 全部 + 团队 Key 共享 + 用量审计 + 配置统一下发 | ¥199-299/座位/月 |
| **Cloud Bridge Tokens** | 任意档位可加购 | 在工作台中开启逐梦云路由,按 token 计费 | 沿用现 token 价 |
| **Enterprise** | 大客户 | 私有化部署 + SLA + 定制 + 专属服务 | 商务洽谈 |

### 3.3 与现有 sub2api 业务的兼容

现有的 sub2api 网关、token 充值、API 控制台**全部保留**,作为 Cloud Bridge 的服务端。差异只在叙事:

- 之前:用户直接对接 sub2api API,平台主要靠 Token 差价收入
- 现在:用户对接本地工作台,工作台再(可选)对接 sub2api 云端;平台靠"软件订阅 + Token 差价"

**存量用户迁移路径:**

1. 现 sub2api 个人 API 用户 → 引导下载工作台,体验本地路由,逐步过渡到 Pro 订阅
2. 现 sub2api 团队用户 → 优先转 Workstation Team,Cloud Bridge 默认开启
3. 现存企业大客户 → 升级谈私有化方案

## 4. 合规姿态的本质变化

这一点单独拎出来讲,因为它是 v2 战略最关键的杠杆。

### 4.1 v1 状态下的合规风险

- 平台运营一个 API 服务,提供未在国内备案的境外模型调用
- 用户数据(prompt、completion)经过平台服务器
- 法律定性接近"未备案的网络信息服务提供者"
- 哪怕加了"仅向境外用户提供服务"的协议条款,灰度依然存在

### 4.2 v2 状态下的合规姿态

- 平台主要业务变成"销售桌面软件" —— 类似一个 IDE、一个客户端工具
- 用户的 key 在用户本地,用户的数据流经用户自己的网络
- 平台扮演的是"工具供应商"角色,与 OpenAI、Anthropic 没有直接资金或数据关系
- Cloud Bridge 是一个**可选**的增值服务,且使用者已经主动确认走平台云端

**实际效果:** 监管视角下,我们更像"做软件的"而不是"做中转的"。这给了我们以下空间:

- 主品牌可以更明确地面向国内市场推广(强调"软件 + 工具"维度,弱化"API 中转")
- 企业销售可以正面打私有化部署,完全规避数据出境问题
- 个人用户教育门槛降低:"装个软件"远比"理解什么是 API 中转"更容易

## 5. 差异化:为什么不是 LiteLLM / OpenRouter / Cursor

这是个会被反复问到的问题,值得提前想清楚。

### 5.1 vs LiteLLM(开源 Python 路由库)
- LiteLLM 是 Python 库,需要用户写代码或起 Docker
- 工作台是桌面 App,零代码,GUI 配置
- LiteLLM 不带工具,只是路由层
- 工作台带 Augment / Codex / DeepRotor 等已魔改的旗舰工具

### 5.2 vs OpenRouter(云端 API 聚合)
- OpenRouter 是云服务,key 在平台,数据经过平台
- 工作台是本地软件,key 在用户,数据可以完全不出本地
- OpenRouter 不带工具,只是 token 转售
- 工作台带工具且统一桌面体验

### 5.3 vs Cursor(独立 IDE)
- Cursor 是单一工具(IDE),用户每次调用模型都经过 Cursor 服务器
- 工作台是工具集合 + 本地网关,Cursor 类工具只是其中之一
- 用户在工作台内用 Cursor 体验的工具(Augment Code),但模型路由完全自主可控

### 5.4 vs Continue.dev(开源 VS Code 插件)
- Continue 是 VS Code 插件,锁在 VS Code 内
- 工作台跨 IDE、跨工具,Augment / Codex / Claude Code 都能挂载
- Continue 没有团队/企业管理能力,工作台有

**核心差异化:**

> 市面上没有第二家公司在做"桌面 App + 本地网关 + 多工具集成 + BYOK + 可选云端"这个全栈组合。

## 8. 待解决的问题(留待最终拍板)
