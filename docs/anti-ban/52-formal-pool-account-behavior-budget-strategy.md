# 正式号池账号行为安全预算策略（第一版）

日期：2026-06-14  
状态：阶段四设计草案，建议先观察模式落地，不直接硬拦生产流量  
适用范围：Sub2API + CC Gateway 联合部署下的 Anthropic 正式订阅号池  
数据来源：生产机只读查询、`usage_logs`、`ops_error_logs`、账号状态字段、`session-budget.jsonl` 安全摘要  

## 0. 执行结论

正式号池需要“账号行为安全预算”，但第一版不建议直接把预算做成强硬拦截。推荐路线是：

1. 继续使用观察模式记录账号、用户、会话、模型、出口代理、5 小时窗口、7 天窗口、限流和认证失败。
2. 调度侧先做软控制：调整账号权重、排队、降速、冷却和换号，不直接破坏正常 Claude Code 工具循环。
3. 明确危险状态才硬拦：认证失败、权限/风控、代理错配、身份边界失败、429 后继续重试、运行证据缺失且无法恢复。
4. 两套策略并存：
   - **平滑策略**：7 天内温和、平均地消耗周额度的 95%-100%。
   - **激进策略**：在安全边界内尽快合理消耗周额度，建议按 3 天目标曲线设计。

预算的目标不是“少用”，而是在真实用户请求存在时，让订阅账号尽量充分使用，同时避免撞限流、风控、登录态失效和异常请求形态。

本次查询了 429 发生前的 5 小时和 7 天窗口。典型样本：

| 账号 | 429 类型 | 429 前 5 小时内部成本 | 429 前 7 天内部成本 | 判断 |
|---|---|---:|---:|---|
| `anthropic-setup-204.1.108.104-扫` | 普通账号限流 | 217.904100 | 878.173405 | 明显进入高风险用量区，后续还出现 401 |
| `疑似限额-anthropic-setup-207.97.155.20-扫` | 普通账号限流 | 79.196519 - 131.532384 | 726.916594 - 823.670347 | 多次 429，说明不能只看 RPM |
| `anthropic-setup-207.97.139.47` | 普通账号限流 | 152.677201 | 649.788645 | 5 小时窗口压力明显 |
| `anthropic-setup-207.97.139.47` | 长上下文额度提示 | 36.180085 | 518.973540 | 长上下文有独立风险，不完全等同普通 tokens 成本 |

判断：429 不是单一 RPM、tokens 或内部美元可以解释的结果。它同时受 5 小时窗口、7 天窗口、模型、长上下文、账号状态和上游动态策略影响。

## 3. 已有 session-budget ledger 能力

生产机已有 `session-budget.jsonl`，安全摘要字段已经覆盖：

- 会话：模型族、消息数、工具定义数、工具调用数、工具结果数、thinking、stream、body 大小桶、输出配置形态、验证结果、fallback 结果。
- 账号：5 小时状态、5 小时百分比桶、5 小时重置桶、7 天状态、7 天百分比桶、7 天重置桶、冷却、隔离、代理引用、调度权重。
- 用户：活跃会话数桶、风险分桶、使用份额桶。
- 池策略：目标天数、目标利用率桶、状态、动作、推荐权重、队列优先级、是否硬拦。
- 决策：观察模式、动作、原因码、安全摘要。

最近 ledger 统计显示：

- 大多数决策仍是观察模式。
- 已能区分 `catch_up`、`maintain`、`slow_down`、`cooldown`。
- 已能区分 5 小时和 7 天利用率桶。
- 已出现 P0 身份边界失败、控制面模型策略、代理错配等风险事件。

这说明第一版账号预算不必从零开始，可以在现有 ledger 基础上做调度软控制和 UI 中文解释。

## 4. 两种策略共同底线

平滑策略和激进策略共用同一套安全底线。激进策略只能提高合规流量的调度权重，不能降低安全标准。

### 4.1 必须暂停账号调度的状态

以下情况应直接暂停该账号进入新请求调度：

1. 上游返回 401：登录态失效、认证失败、令牌不可用。
2. 上游返回 403：权限、风控、订阅不可用、访问被拒。
3. 代理出口与账号绑定不一致。
4. 运行时注册映射缺失，且无法通过诊断面板安全恢复。
5. 账号健康检查明确失败。
6. 出现账号级风险文本、异常活动、KYC 或类似风控信号。
7. 429 后仍继续对同账号重试。
8. 请求身份边界失败、验证失败、fallback 异常或最终请求形态被篡改。

### 4.2 429 处理

429 不是“多试几次就会好”。推荐动作：

1. 账号立刻进入冷却，不再接新请求。
2. 如果上游提供重置时间，等待对应窗口恢复。
3. 如果没有重置时间，保守按 5 小时窗口冷却。
4. 冷却期间不做连续真实健康检查。
5. 冷却后先低权重恢复，再观察 5 小时和 7 天状态。

### 4.3 401 / 403 处理

- 401：优先判断登录态或访问令牌失效，需要刷新认证；刷新失败则保持错误状态。
- 403：优先按权限、风控、订阅能力、模型权益或账号状态异常处理；不应自动反复重试。
- 401 / 403 不能被“激进策略”绕过。

## 5. 平滑策略

平滑策略目标：7 天内温和、平均地消耗周额度的 95%-100%。

### 5.1 消耗曲线

7 天约有 33.6 个 5 小时窗口。

```text
目标每 5 小时消耗 = 周额度的 95%-100% / 33.6
约等于每 5 小时 2.8%-3.0%
```

目标进度：

| 时间 | 目标 7 天累计利用率 |
|---|---:|
| 第 1 天 | 约 14% |
| 第 2 天 | 约 28% |
| 第 3 天 | 约 42% |
| 第 4 天 | 约 56% |
| 第 5 天 | 约 70% |
| 第 6 天 | 约 84% |
| 第 7 天 | 95%-100% |

### 5.2 推荐初始参数

| 项 | 建议值 |
|---|---:|
| 单账号有效 RPM | 12 起步，健康时允许到 17 |
| 单账号活跃会话 | 4-5 |
| 单账号同时上游请求 | 2-3 |
| 正常调度权重 | 1.0 |
| 落后目标权重 | 1.2 |
| 超前目标权重 | 0.8 |

说明：这里的“同时上游请求”是指同一时刻正在请求上游的数量；“活跃会话”是指一个账号可承接的不同用户/任务会话数量。一个会话不等于一个并发请求。

### 5.3 5 小时窗口动作

| 5 小时利用率 | 动作 |
|---|---|
| 0%-70% | 正常调度 |
| 70%-85% | 降低追赶，不主动增加压力 |
| 85%-95% | 不主动派新高成本会话和长上下文任务 |
| 95%-100% | 不开新会话，只允许安全收尾 |
| 429 | 立即冷却 |

### 5.4 7 天窗口动作

| 7 天利用率状态 | 动作 |
|---|---|
| 低于目标曲线 5% 以上 | 轻微追赶 |
| 接近目标曲线 | 正常 |
| 高于目标曲线 5% 以上 | 降速 |
| 超过 95% | 只保留必要生产流量 |
| 100% 或出现 429 | 冷却或等待窗口恢复 |

## 6. 激进策略

激进策略目标：在安全边界内尽快合理消耗完周额度，建议按 3 天消耗 95%-100% 设计。

激进策略不是空跑烧额度，也不是绕过冷却。它只是在真实用户请求存在时，更积极地把合规流量调度给该账号。

### 6.1 消耗曲线

3 天约有 14.4 个 5 小时窗口。

```text
目标每 5 小时消耗 = 周额度的 95%-100% / 14.4
约等于每 5 小时 6.6%-6.9%
```

目标进度：

| 时间 | 目标 7 天累计利用率 |
|---|---:|
| 第 1 天 | 约 32%-33% |
| 第 2 天 | 约 63%-67% |
| 第 3 天 | 95%-100% |

### 6.2 推荐初始参数

| 项 | 建议值 |
|---|---:|
| 单账号有效 RPM | 17 作为主档 |
| 健康且明显落后时短时 RPM | 最高 25，需自动回落 |
| 单账号活跃会话 | 6-8 |
| 单账号同时上游请求 | 4-6 |
| 正常调度权重 | 1.0 |
| 落后目标权重 | 1.8 |
| 超前目标权重 | 0.65 |

生产历史里三号出现过 32、37、47 的短时 RPM 峰值，但同时也出现过 429。因此 25 以上不建议作为持续配置，只能作为后续灰度观察后的短时弹性上限。

### 6.3 5 小时窗口动作

| 5 小时利用率 | 动作 |
|---|---|
| 0%-60%，且 7 天落后 | 提高调度权重 |
| 60%-80% | 正常激进 |
| 80%-90% | 降低高成本模型、长上下文和新会话优先级 |
| 90%-95% | 不开新重任务 |
| 95%-100% | 停止新会话，只允许安全收尾 |
| 429 | 立即冷却 |

### 6.4 7 天窗口动作

| 7 天利用率状态 | 动作 |
|---|---|
| 明显落后 3 天曲线 | 提高权重，优先承接真实用户任务 |
| 接近 3 天曲线 | 维持激进主档 |
| 超过曲线 | 降权，避免撞 5 小时窗口 |
| 超过 95% | 停止主动追赶 |
| 100% 或出现 429 | 冷却或等待恢复 |

## 7. 上游窗口信息缺失时的保守 fallback

如果上游没有返回 5 小时和 7 天窗口信息，不能盲目激进。内部成本只能作为保守参考。

| 指标 | 注意线 | 降速线 | 禁止激进线 |
|---|---:|---:|---:|
| 5 小时内部成本 | 80 | 120 | 150 |
| 7 天内部成本 | 650 | 800 | 900 |

这些不是硬封线，因为生产中存在 7 天内部成本超过 900 仍可调度的账号。真正调度应以上游窗口状态、账号健康和错误类型为准。

## 8. 多用户共享账号时的公平性

当 5-10 个账号承接 20 个用户请求时，安全预算还需要防止单个用户拖垮共享账号。

建议：

1. 每个账号的并发请求按活跃用户做加权公平分配。
2. 单用户短时间触达过多账号时降低该用户调度优先级。
3. 单用户 429 后立即重试，视为用户侧风险，先排队或降速。
4. 高成本模型、长上下文、长输出应计入用户份额。
5. 同一个正在流式输出的会话不做中途迁移；只在安全边界点换账号。
6. 账号级 401 / 403 / 风控时隔离整个账号，不只隔离某个用户。

## 9. 诊断面板中文解释

运营界面不要直接暴露英文桶名。建议中文化：

| 内部状态 | 中文解释 |
|---|---|
| `observe_pool_catch_up` | 当前账号用量低于目标，系统建议适当增加调度 |
| `observe_pool_on_track` | 当前账号用量符合目标，维持正常调度 |
| `observe_pool_slow_down` | 当前账号用量高于目标，系统建议降速 |
| `cooldown_recommendation` | 上游限流或窗口接近上限，账号正在冷却 |
| `observe_conservative_missing_utilization` | 未拿到完整 5 小时/7 天窗口信息，按保守策略调度 |
| `p0_identity_boundary_fail` | 请求身份边界失败，已拒绝调度 |
| `proxy_mismatch` | 代理出口与账号绑定不一致 |
| `runtime_evidence_missing` | 缺少运行证据，需要重新注册映射或执行安全修复 |
| `401` | 登录态或认证失效，需要刷新账号认证 |
| `403` | 权限、订阅能力或风控异常，需要人工复核 |
| `429` | 上游限流，等待 5 小时或 7 天窗口恢复 |

## 10. 第一版落地建议

第一版不要直接把预算做成强硬生产开关。建议分三步：

### 第一步：观察模式增强

- 继续记录 5 小时和 7 天窗口桶。
- 记录账号、用户、会话、模型、出口代理、错误类型。
- 在看板显示中文解释和推荐动作。
- 不改变真实调度结果，除非 P0 硬风险。

### 第二步：软调度

- 平滑策略默认启用。
- 激进策略只对明确标记为 production 且健康的账号启用。
- 只调整权重、队列优先级、是否开新会话。
- 429 后冷却、401/403 暂停账号调度。

### 第三步：灰度硬保护

- 对单账号、单用户、小流量分组灰度。
- 只把明确危险状态升级为硬拦。
- 保留总开关和快速回滚。
- 每次策略变更后观察 24 小时。

## 11. 与部署顺序的关系

阶段一到三的 native CLI 请求形态增强可以先部署。阶段四预算建议独立推进，不阻塞当前 Sub2API + CC Gateway 成对部署。

推荐部署顺序：

1. 备份当前生产镜像、配置和数据库。
2. 先部署新版 CC Gateway。
3. 健康检查通过后部署新版 Sub2API。
4. 做本机 Claude Code CLI 冒烟测试。
5. 检查号池看板、三个重点账号、最近请求、诊断面板。
6. 阶段四预算先以观察模式继续积累数据，再进入软调度。

## 12. 当前不建议做的事

1. 不建议用内部美元成本代替官方 5 小时/7 天窗口。
2. 不建议把 30+ RPM 峰值当成持续安全配置。
3. 不建议激进策略绕过 429 冷却。
4. 不建议在窗口信息缺失时继续激进追赶。
5. 不建议对正常 Claude Code 的 tools、thinking、stream、长上下文做低阈值硬拦。
6. 不建议为了打满额度制造空跑请求。
