## `d662c9730` Claude Fable 5

### What upstream changes

Upstream adds `claude-fable-5` to:

- Backend default Antigravity model mapping and Bedrock default mapping.
- Antigravity model list and Claude model metadata exposure.
- Claude identity/model constants.
- Bedrock model compatibility helpers for Fable-specific thinking behavior.
- Frontend model whitelist, preset mappings, account usage/status labels, and key-use config generation.

Notable upstream assumptions:

- Fable 5 has a 1,048,576-token context window and 128,000 output-token limit in generated client config.
- Fable 5 uses `thinking.type = adaptive` without `budget_tokens` in the generated config.
- Bedrock Fable mapping is `anthropic.claude-fable-5`, without the same regional prefix shape as existing Claude Bedrock models.

### Direct value

- High value for Claude Code / Claude pool operators if upstream Fable 5 is available and stable.
- Potentially useful for Codex Desktop only if we intentionally expose Claude-family models through a compatible route or model mapping.
- No direct value for DeepSeek or AGNES routing, except avoiding accidental wildcard misrouting when users request the new model.

### Conflict risk with local architecture

- Local Antigravity mapping already contains deliberate migration rules such as `claude-sonnet-4-5-thinking -> claude-sonnet-4-6` and Opus aliases. Blindly inserting Fable could interact with wildcard mappings (`claude-*`) and model allowlists.
- Local Claude anti-ban logic depends on beta/header/persona behavior. Fable enablement must not change existing Claude Code mimicry defaults or FormalPool account selection.
- Bedrock Fable assumptions need live validation. The upstream model ID shape and always-on adaptive thinking behavior should not be accepted without AWS smoke.
- Frontend whitelist exposure can cause users to select Fable before the backend, account pool, and pricing are ready.

### Recommendation

Do not cherry-pick `d662c9730` as-is. Treat it as a spec/input and implement a local Fable enablement project.

Default exposure should be off until live upstream smoke passes. Add exact model support first in backend/account capability detection, then enable frontend selection.

### Claude Fable 5 enablement plan

1. Claude account pool support
   - Add explicit `claude-fable-5` capability handling for Claude/Anthropic accounts.
   - Ensure Claude pool and FormalPool scheduling can include or exclude Fable per account, not via a broad `claude-*` wildcard only.
   - Verify failover does not downgrade Fable into Sonnet/Opus unless explicitly configured.
   - Add account-level and group/channel-level tests for allowlist, mapping, fallback, and usage attribution.

2. Antigravity / Bedrock / Claude constants
   - Add `claude-fable-5` to model constants only after backend capability tests exist.
   - Antigravity: add to `DefaultModels`, default mapping, and identity prompt metadata; verify request transformer thinking behavior with fixtures.
   - Bedrock: add a guarded mapping only after live AWS model ID validation. Do not assume `anthropic.claude-fable-5` is accepted in every region.
   - Claude constants: add identity metadata without changing existing Claude Code beta/header defaults.

3. Frontend model whitelist
   - Add to `frontend/src/composables/useModelWhitelist.ts` after backend is ready.
   - Add preset mappings for Anthropic, Antigravity, and Bedrock, but gate them behind a backend-supported model list if possible.
   - Update account status/usage short labels only after backend metrics report the model consistently.

4. Pricing and context window
   - Confirm pricing before adding billing defaults.
   - Treat upstream's 1,048,576 context / 128,000 output numbers as provisional until live upstream or provider docs confirm them.
   - Add explicit usage-log tests so Fable billing is not silently counted as a fallback model.

5. Codex Desktop exposure
   - Do not expose Fable to Codex Desktop by default.
   - If needed, expose through an opt-in mapping or group/channel config so Codex-specific GPT/Codex model routes remain untouched.
   - Verify no interaction with Codex Gateway model aliases, Responses WebSocket v2, or prompt cache key logic.

6. Thinking variant
   - Upstream adds only `claude-fable-5`, not a separate `claude-fable-5-thinking` ID.
   - Treat Fable as an adaptive-thinking model only after live upstream confirms request shape.
   - For Bedrock, test that `thinking.type=adaptive` with no `budget_tokens` is accepted and that `enabled` is rejected or transformed correctly.

7. Required live upstream smoke
   - Anthropic/Claude direct: non-stream text, stream text, tool use, cache_control, and thinking/adaptive request.
   - Claude Code pool/FormalPool: real Claude Code request with local anti-ban headers/persona unchanged.
   - Antigravity: `/v1/models`, non-stream, stream, tool-use, and thinking fixtures.
   - Bedrock: per-region model ID resolution, non-stream, stream, cache_control ttl, context_management beta, and adaptive thinking.
   - Frontend: model appears only when backend supports it; generated config uses the expected context/output limits.

## `bf28a0098` and `12962bab2` Bedrock unsupported fields/header filtering

### What upstream changes

`bf28a0098`:

- Deletes unsupported Bedrock top-level request body fields such as `provider` and `metadata`.
- Deletes stale `anthropic_beta` body field when final filtered beta list is empty.
- Adds `context-management-2025-06-27` and `fine-grained-tool-streaming-2025-05-14` to Bedrock beta whitelist.

`12962bab2`:

- Changes `ApplyBedrockCCCompat` signature from `context.Context` to `*gin.Context`.
- Moves `anthropic-beta` request header filtering into `ApplyBedrockCCCompat`.
- Removes a separate handler-side filtering call in favor of one body+header compat function.

### Direct value

- Valuable for Bedrock/Claude Code compatibility, especially avoiding Bedrock 400s from unsupported top-level fields or beta leakage.
- No direct value for DeepSeek or AGNES.
- Potential value for Claude Code users routed to Bedrock, but only if it preserves local anti-ban and CCH semantics.

### Existing local state

Local code already has a channel-level Bedrock CC compat path:

- `GatewayService.ApplyBedrockCCCompat(ctx, body, model, account, groupID)`.
- `sanitizeBedrockCCFields`, `sanitizeBedrockThinking`, `sanitizeBedrockToolUseIDs`, and `sanitizeBedrockCCBetaTokens`.
- Bedrock request preparation with `PrepareBedrockRequestBodyWithTokens` and beta filtering.

Local code does not currently include the upstream `provider`/`metadata` deletion in `PrepareBedrockRequestBodyWithTokens`, nor the `ApplyBedrockCCCompat(*gin.Context, ...)` header filtering refactor.

### Conflict risk with local architecture

- Header filtering is sensitive: local Claude/CC anti-ban work computes final Anthropic beta headers, signs CCH after body sanitization, and preserves strict passthrough/mimicry cases. A broad signature change that edits request headers inside compat code could affect shared Gateway behavior.
- Bedrock CC compat is channel-flagged locally. Any new behavior should stay inside that channel/platform scope and not leak into non-Bedrock Claude/CC Gateway paths.
- Adding beta tokens based only on upstream claims should be verified against live Bedrock. Incorrect whitelist expansion can leak unsupported tokens and create 400s.

### Recommendation

Do not cherry-pick the combined commits as-is.

Preferred local rewrite:

1. Safe body cleanup patch
   - In `PrepareBedrockRequestBodyWithTokens`, delete `provider` and `metadata`.
   - If filtered beta tokens are empty, delete existing `anthropic_beta`.
   - Add unit tests for both behaviors.

2. Beta whitelist patch behind validation
   - Add `context-management-2025-06-27` and `fine-grained-tool-streaming-2025-05-14` only after live AWS smoke confirms support for the target model/region.
   - If uncertain, keep behind a Bedrock beta allowlist setting default off.

3. Header filtering patch only if needed
   - Avoid changing `ApplyBedrockCCCompat` to take `*gin.Context` unless a failing test proves header leakage.
   - If implemented, isolate it to Bedrock CC compat and test that non-Bedrock Claude, CC Gateway, FormalPool, and strict passthrough headers remain unchanged.

## CP5 deferred-topic absorption follow-up

Date: 2026-06-11
Branch: `codex/selective-v0.1.136`

This follow-up records the post-review absorption decisions for the topics that were previously skipped or deferred. The guiding rule remains: absorb concrete runtime value when it is narrow and testable; avoid importing upstream control-plane behavior that can lock operators out or weaken local Claude Code / CC Gateway safety.

### Bedrock CC compat shape guard

A review found that the selective branch had a shape mismatch for `features_config.bedrock_cc_compat`:

- backend expected a channel-scoped boolean;
- the admin UI save path still wrote a platform map;
- the admin UI read path only recognized the boolean.

Resolution:

- keep the canonical shape as a channel-scoped boolean;
- make the backend tolerate old map-shaped values by treating any explicit `true` as enabled;
- make the UI read old map-shaped values correctly;
- make the UI save only the canonical boolean shape.

This is a protection fix for the local Claude Code / Bedrock compat toggle and should be kept before any merge back to main.

### Bedrock unsupported fields and beta filtering

Absorbed as a local narrow patch, not as the upstream header-refactor commit:

- remove unsupported top-level `provider` and `metadata` fields in the direct Bedrock request body preparation path;
- delete stale `anthropic_beta` when the final filtered Bedrock token list is empty;
- add Bedrock beta allowlist support for `context-management-2025-06-27` and `fine-grained-tool-streaming-2025-05-14`.

Still deliberately not absorbed:

- changing `ApplyBedrockCCCompat` to take `*gin.Context`;
- mutating the inbound `anthropic-beta` request header in the shared Gateway path.

Reason: the local code has Claude Code mimicry, FormalPool, CC Gateway, Antigravity, and strict passthrough semantics that depend on not globally rewriting request headers before route/account decisions.

### Claude Fable 5

Absorbed as a local enablement patch instead of cherry-picking upstream wholesale.

Backend additions:

- Claude and Antigravity model catalogs include `claude-fable-5`;
- Antigravity default mapping includes exact passthrough;
- persisted Antigravity mappings are backfilled with a non-destructive migration;
- existing Antigravity explicit mapping accounts auto-fill Fable passthrough at runtime;
- Bedrock mapping uses official `anthropic.claude-fable-5`;
- Bedrock Fable thinking sanitization uses adaptive thinking and removes `budget_tokens`;
- Codex Gateway registers Fable as an explicit opt-in Anthropic direct model with 1,048,576 context and 128,000 output, but it is not added to the default enabled model list;
- billing/pricing has explicit Fable pricing so manual enablement does not silently fall back to Sonnet pricing.

Frontend additions:

- Claude, Antigravity, and Bedrock admin model lists include Fable;
- account status/usage display knows Fable;
- OpenCode Antigravity Claude config emits Fable with adaptive thinking and no budget token field.

Safety decisions:

- no invented `claude-fable-5-thinking` model slug;
- no `claude_code_max` Fable variant;
- no FormalPool healthcheck change;
- no automatic Codex Desktop exposure through default enabled models.

### Scheduler debug log loop optimization

Absorbed as a narrow performance patch. Only the per-account debug-detail logging loops inside `listSchedulableAccounts` are guarded by `slog.Default().Enabled(ctx, slog.LevelDebug)`. No account query, filtering, scheduling, session budget, sticky-session, or routing logic was changed.

### Admin compliance gate

Not absorbed as runtime code in this branch.

The upstream feature has product/legal value, but it is not a gateway/runtime correctness fix and it touches high-risk local areas: admin routes, generated wiring, frontend shell/router/API client, and legal-document build plumbing. Direct absorption could lock operators out of the admin UI or break Docker frontend builds because upstream imports root-level legal docs from the frontend bundle.

Local absorption decision:

- keep the default-off design and recovery-path requirements documented here;
- implement later as a separate feature only if explicitly requested;
- requirements for any future implementation: feature flag default off, fail-open on setting/database errors, explicit recovery routes, no gating of `/v1/**`, `/codex/v1/**`, `/backend-api/codex/**`, payment webhooks, or setup flow, and local legal copy review before shipping text.
