# 逐梦 Agent 数据资产工作台设计方案

> 版本：v0.1（第三份文档·讨论稿）
> 日期：2026-06-18
> 适用范围：逐梦 Agent 个人版、团队版、企业/私有化版；sub2api / gateway；本地数据资产工作台；客户侧私有数据服务
> 文档定位：**产品方案 + 数据资产化架构 + 清洗/整理/贡献/交易机制**，不含实现代码
> 配套文档：
> - [文档1：逐梦 Agent 训练数据飞轮设计方案](./zhumeng-agent-training-data-flywheel-design.md)

## 0. 一句话结论

逐梦 Agent 不应只是“接管 Codex / Claude Code 的 Agent 工具”，也不应只是“采集训练数据”。它应进一步把个人、团队和企业在 Agent 使用过程中沉淀的行为、工具、workflow、skill、产物、反馈和结果，整理成客户真正能看懂、能复用、能优化、能训练、能贡献、能交易的 **AI 数据资产**。

核心链路：

```text
Agent 使用数据
  -> 数据资产工作台
  -> Workflow 优化
  -> Skill / Eval / 训练数据
  -> 数据贡献与交易
  -> 逐梦共建池 / 公共训练池
```

逐梦的竞争力不是单个去重器、脱敏器、标注器或 dashboard，而是：

> 用开源部件做底盘，用公开方法论做参考，用逐梦 Agent 的数据结构、主权边界和产品闭环做升华，把个人/企业的 Agent 使用数据变成可看懂、可优化、可训练、可贡献、可交易的 AI 数据资产。

## 2. 核心产品定位

### 2.1 不是“日志中心”，而是“数据资产工作台”

用户不会因为我们保存了很多 trace 而付费。用户真正需要的是：

- 我到底积累了哪些 AI 工作数据？
- 哪些数据有价值，哪些只是噪声？
- 哪些 workflow 反复出现，值得标准化？
- 哪些 skill 是可以沉淀为个人/组织资产的？
- 哪些样本能变成私有 eval？
- 哪些样本能用于微调、偏好训练、工具调用训练？
- 哪些数据可以贡献给逐梦，换权益、收益或共建能力？
- 哪些数据绝不能出域？

所以产品入口不应叫“训练日志”或“trace 管理”，而应叫：

```text
数据资产工作台
AI 工作资产
个人/企业 AI 记忆
Workflow / Skill 资产库
训练数据准备区
数据贡献与交易区
```

### 2.2 两个核心功能

本文将产品能力分为两条主线。

#### A. 数据价值再利用

面向个人、团队和企业，让客户自有池中的数据产生价值：

- workflow 盘点；
- 高频任务聚类；
- 失败流程识别；
- token-maxing 与成本归因；
- skill 候选生成；
- 私有 eval 候选；
- 私有训练数据候选；
- 个人可携带 AI 工作记忆；
- 企业 ROI / 治理 / 流程优化看板。

#### B. 数据贡献与交易

面向愿意共享或变现数据的客户：

- 生成可预览数据包；
- 分级、脱敏、质量评分；
- 估值与对价建议；
- 客户确认用途、期限、收益和撤回规则；
- 写入出境账本与交易账本；
- 进入逐梦共建池；
- 再筛选进入公共训练池或行业 benchmark。

## 3. 三池模型在本文中的使用

本文继续沿用文档1/2 的三池模型。

| 数据池 | 默认归属 | 本文中的用途 |
|---|---|---|
| 客户自有池 | 个人 / 团队 / 企业租户 | 数据整理、workflow 优化、skill 沉淀、私有 eval、私有训练 |
| 逐梦共建池 | 客户显式 opt-in / 企业合同化贡献 | 产品改进、行业 benchmark、共建模型、数据交易、逐梦购买数据 |
| 公共训练池 | 从逐梦共建池筛选进入 | 逐梦自有模型训练、通用/行业能力增强 |

关键原则：

1. **客户自有池默认不等于逐梦可用。** 个人和企业开启数据飞轮后，默认只进入个人/租户自有池。
2. **逐梦共建池必须显式授权。** 个人 opt-in、团队 opt-in、企业合同化贡献，均需 `contribution_consent_id`。
3. **公共训练池不能直接接收客户数据。** 必须从共建池中再经过授权、脱敏、质量、license、teacher_eligible 检查后进入。
4. **数据贡献与交易只发生在客户可预览、可定价、可限定用途之后。** 不允许“静默把日志拿去卖/训”。

## 4. 本地、客户侧与云端的分工

数据资产工作台可以在用户本机看到，但计算不必全部在本机完成。推荐架构是：

```text
local-first / tenant-first / cloud-optional
```

### 4.1 个人用户

个人用户默认本地处理：

```text
本机采集
  -> 本机分级
  -> 本机脱敏
  -> 本机清洗
  -> 本机生成个人工作流洞察 / skill / 数据包预览
```

云端只在用户明确开启时参与：

- 多设备同步；
- 云端备份；
- 更强模型辅助整理；
- 数据贡献/交易；
- 逐梦代整理训练包。

自动化建议：

| 触发 | 处理内容 |
|---|---|
| 每个任务结束后 | 轻量 episode 整理、基础标签、隐式反馈 |
| 每天 | 去重、任务分类、敏感扫描、摘要 |
| 每周 | workflow 聚类、个人 skill 候选、训练候选、贡献包候选 |
| 手动触发 | 生成训练包、贡献包、交易包、导出包 |

### 4.2 企业用户

企业用户建议采用客户侧集中处理：

```text
员工本机 Agent
  -> 本机初步分级/脱敏
  -> 企业内网/私有化数据资产服务
  -> 企业管理员工作台
```

默认分工：

| 层级 | 职责 |
|---|---|
| 员工本机 | 采集、初步分级、敏感检测、上传前出境决策 |
| 企业侧服务 | 统一清洗、聚类、workflow 挖掘、skill 候选、私有 eval、团队看板 |
| 逐梦云端 | 默认不参与；仅在合同允许时做共建、交易、模型训练增强 |

企业自动化建议：

| 周期 | 处理内容 |
|---|---|
| 实时/准实时 | L0/L1 指标、任务状态、基础事件聚合 |
| 每晚 | 增量清洗、去重、质量评分、敏感扫描 |
| 每周 | workflow 聚类、skill 候选、团队/角色/流程洞察 |
| 每月 | ROI 报告、私有 eval 更新、训练候选包、治理报告 |
| 手动触发 | 数据包导出、私有训练、贡献逐梦、交易授权 |

### 4.3 云端可选增强

逐梦云端适合处理以下任务：

- 客户授权后的大规模二次清洗；
- 数据包估值；
- 跨客户聚合 benchmark；
- 贡献数据的二次安全审计；
- 公共训练池筛选；
- 逐梦模型训练；
- 行业数据共建。

云端不应默认处理：

- 企业业务原文；
- 未授权 L2/L3/L3+；
- 未经预览的数据贡献包；
- 人员级行为画像；
- hidden/internal CoT。

## 5. 两条清洗流水线

数据资产工作台必须区分两条不同目的的流水线。

### 5.1 Workflow 改进流水线

目标：帮客户优化工作方式，而不是训练模型。

输入：

- episode；
- tool call / tool result；
- 用户采纳/修改/回滚；
- 任务结果；
- 成本、token、耗时；
- workflow / skill / MCP / plugin 使用情况；
- 团队/角色/项目上下文。

处理：

```text
原始 episode
  -> 任务分类
  -> workflow 聚类
  -> 高频流程识别
  -> 失败点识别
  -> 成本/token 归因
  -> 人工步骤识别
  -> 自动化机会识别
  -> skill 候选生成
  -> workflow 优化建议
```

输出：

- 高频任务 Top N；
- 失败流程 Top N；
- token 消耗异常；
- 可自动化流程；
- 可沉淀 skill；
- 推荐 workflow 模板；
- 模型路由建议；
- 团队/角色/项目聚合洞察；
- ROI / 节省时间估算。

默认隐私姿态：

- 企业默认团队/角色/项目/流程聚合；
- 人员级排名、个人失败率、个人 token 异常需要租户显式开启；
- 个人用户默认只给本人看。

### 5.2 模型训练数据流水线

目标：把客户授权数据变成可训练样本。

输入：

- L2/L3/L3+ episode；
- prompt / response；
- tool trajectory；
- before/after human edit；
- accept/reject；
- outcome；
- reasoning trace；
- 模型身份与 license；
- consent / contribution_consent。

处理：

```text
原始 episode
  -> 权属/授权检查
  -> PII/secret/license 扫描
  -> 去重
  -> 去低质量样本
  -> 去 eval 污染
  -> 质量评分
  -> 构造 SFT 样本
  -> 构造 DPO pair
  -> 构造 tool-use trajectory
  -> 构造 router dataset
  -> 构造 reward / RL transition 候选
```

输出：

- private eval；
- SFT JSONL；
- DPO pair；
- tool trajectory；
- router dataset；
- reward model 数据；
- RL transition 候选；
- teacher trajectory 候选；
- 训练准备度报告。

默认隐私姿态：

- 客户自有池中的训练包只归客户；
- 逐梦共建训练包必须另行授权；
- L3/L3+ 原文和 CoT 默认不交易；
- 公共训练池必须二次筛选。

## 6. 数据贡献与交易机制

### 6.1 贡献不是上传日志，而是生成数据包

数据贡献/交易的最小单位不是 raw log，而是客户可预览的数据包：

```text
dataset_package
```

数据包应包含：

- 数据范围；
- 数据档级：L0/L1/L2/L3/L3+；
- 样本数量；
- 任务类型；
- 行业/领域标签；
- 脱敏状态；
- 质量评分；
- 风险评分；
- 可用用途；
- 禁止用途；
- 授权期限；
- 价格/权益建议；
- 撤回/删除规则。

### 6.2 交易流程

```text
客户选择贡献范围
  -> 生成数据包预览
  -> 分级/脱敏/质量评分
  -> 标注可用途
  -> 估值
  -> 客户确认价格/权益/用途/期限
  -> 签署贡献授权或购买协议
  -> 写入出境账本 + 交易账本
  -> 进入逐梦共建池
  -> 再筛选进入公共训练池 / 行业 benchmark / 共建模型
```

### 6.3 可交易数据类型

| 类型 | 价值 | 风险 | 建议策略 |
|---|---:|---:|---|
| L0 聚合指标 | 中 | 低 | 可规模化交易，必须不可识别单用户/单租户 |
| L1 协议形状 / 工具错误 | 高 | 低 | 适合逐梦收购，用于工具、路由、兼容性优化 |
| L2 脱敏 workflow | 很高 | 中 | 交易主力，需去 PII、商业秘密和第三方个人信息 |
| 经审核 skill/workflow | 很高 | 中 | 明确 IP 归属、授权期限和复用范围 |
| L3 原文轨迹 | 高 | 高 | 默认不交易，仅强合同场景 |
| L3+ CoT / reasoning | 极高 | 极高 | 默认不交易，仅白名单、自托管、强授权、强隔离 |

### 6.4 逐梦的角色

逐梦可以有两个角色：

1. **收购方**：直接购买客户授权数据，用于产品改进、行业 benchmark、模型训练。
2. **撮合方**：未来撮合数据贡献方和模型/行业需求方，但必须先解决合规、授权、收益结算和用途审计。

近期建议先做收购方，不急于做开放数据市场。

## 7. 成熟部件与可借鉴方法

本文不主张从零造所有轮子。行业已有成熟部件和方法论，但没有完整覆盖“Agent 使用数据 -> workflow 优化 -> 训练数据 -> 数据交易”的标准产品。

### 7.1 可借力的开源/公开部件

| 能力 | 可参考/使用部件 | 开放情况 | 用法建议 |
|---|---|---|---|
| 大规模训练数据清洗 | NVIDIA NeMo Curator | 开源，Apache 2.0 | 用于过滤、去重、转换、隐私处理等大规模数据管线参考 |
| 文本数据处理 | Hugging Face DataTrove | 开源，Apache 2.0 | 用于本地/分布式数据处理、过滤、去重参考 |
| 预训练语料构建 | AllenAI Dolma Toolkit | 开源 | 参考语料构建、去重、质量控制 |
| PII 检测/脱敏 | Microsoft Presidio | 开源，MIT | 用于 PII 检测与匿名化参考 |
| 标注审核 | Label Studio | 开源，Apache 2.0 | 可参考标注 UI、审核流、人工质检 |
| Trace 标准 | OpenTelemetry GenAI semantic conventions | 开源标准，Apache 2.0 | 兼容 trace/span/LLM/tool 调用语义 |
| Process mining | PM4Py | 开源，AGPL | 算法参考；商用闭源需谨慎处理许可 |
| 评测体系 | OpenAI Evals / LangSmith 思路 | 公开文档/部分开源或商业 | 参考 trace -> dataset -> eval -> human review 闭环 |
| Task/process mining 产品 | Microsoft Power Automate Process Mining / Task Mining | 商业产品，文档公开 | 学习产品逻辑，不复制实现或 UI |

### 7.2 不能照搬的部分

以下必须自研或重新设计：

- 逐梦 Agent episode schema；
- L0/L1/L2/L3/L3+ 分级；
- 个人/租户自有池；
- 逐梦共建池；
- 公共训练池；
- 出境账本；
- 交易账本；
- workflow/skill 挖掘；
- 数据包估值；
- 客户授权、撤回、删除；
- 用户看得懂的数据资产工作台 UI；
- Agent 行为数据到训练样本的转换逻辑。

### 7.3 原则：借鉴、兼容、升华，不抄袭

```text
公开标准：直接兼容
开源组件：合规使用 / 二次封装
公开产品逻辑：学习方法，重新设计
闭源产品实现：不碰代码，不复制 UI，不复制私有方案
逐梦核心机制：自研
```

## 8. 产品界面建议

用户不应看到复杂的数据工程流水线，而应看到“资产”和“建议”。

### 8.1 工作台四个区

#### 1. 数据健康度

展示：

- 采集了多少任务；
- 各档数据数量；
- 敏感数据比例；
- 高质量数据比例；
- 可用于 workflow 的数据；
- 可用于训练的数据；
- 可贡献/交易的数据；
- 当前风险与缺口。

#### 2. Workflow 洞察

展示：

- 高频任务；
- 失败流程；
- 高 token 消耗流程；
- 可自动化流程；
- 可沉淀 skill；
- 模型路由建议；
- 团队/角色/项目聚合分析。

#### 3. 训练准备度

展示：

- private eval 候选数量；
- SFT 候选数量；
- DPO pair 候选数量；
- tool trajectory 候选数量；
- reward/RL transition 候选数量；
- 缺少哪些类型的数据；
- 哪些数据不能训练以及原因。

#### 4. 贡献与交易

展示：

- 可贡献的数据包；
- 风险等级；
- 预估价值；
- 可获得权益/收益；
- 用途限制；
- 授权期限；
- 撤回规则；
- 是否允许逐梦购买。

### 8.2 个人版与企业版差异

| 能力 | 个人版 | 企业版 |
|---|---|---|
| 默认处理位置 | 本机 local-first | 员工本机 + 企业侧私有服务 |
| 默认视角 | 个人工作流、个人 skill、个人记忆 | 团队/角色/项目/流程聚合 |
| 人员级分析 | 仅本人 | 默认关闭，租户显式开启 |
| 数据贡献 | 个人 opt-in | 合同/管理员策略 |
| 数据交易 | 数据包预览后手动确认 | 合同、账本、审批流 |
| 私有训练 | 个人轻量训练/偏好数据 | 企业私有 eval / 微调 / 行业模型 |

## 9. 自动化策略

数据资产工作台应自动整理，但不能自动贡献、自动交易或自动训练。

### 9.1 自动化等级

| 等级 | 默认 | 内容 |
|---|---|---|
| 自动轻量整理 | 默认开启 | episode 切分、基础标签、去重、敏感扫描、摘要 |
| 自动深度整理 | 默认每周，可关闭 | workflow 聚类、skill 候选、训练候选、质量评分 |
| 自动建议 | 默认开启 | 推荐 workflow、skill、eval、贡献包 |
| 手动确认 | 强制 | 上传 L2/L3、贡献、交易、训练导出、人级分析 |

### 9.2 触发方式

- 任务结束后触发；
- 每日定时任务；
- 每周深度任务；
- 管理员手动运行；
- 数据达到阈值后触发；
- 贡献/交易前强制触发安全扫描。

## 10. 阶段化落地建议

### Phase 1：数据可见

目标：先让客户看到自己积累了什么。

交付：

- 数据健康度；
- L0-L3 分布；
- 本地/租户自有池视图；
- 高频任务；
- 基础敏感扫描；
- 数据可用性标签。

不做：

- 数据交易；
- 自动训练；
- 人级绩效分析；
- 大规模跨客户 benchmark。

### Phase 2：数据可用

目标：让客户能用数据优化工作。

交付：

- workflow 聚类；
- skill 候选；
- 私有 eval 候选；
- 训练候选集；
- 数据清洗 job；
- 质量评分；
- 导出包。

### Phase 3：数据可贡献

目标：让客户可控地贡献数据给逐梦。

交付：

- dataset_package 预览；
- contribution_consent；
- 出境账本；
- 贡献用途限制；
- 逐梦共建池入口；
- 权益/折扣/共建对价。

### Phase 4：数据可交易

目标：让客户数据可以合规变现。

交付：

- 数据包估值；
- 数据购买协议；
- 交易账本；
- 收益结算；
- 行业 benchmark 数据包；
- 逐梦作为收购方的采购流程。

---

## 11. 风险与边界

### 11.1 合规风险

- L2 脱敏不等于无风险，因为 workflow 本身可能是商业 know-how；
- L3/L3+ 原文和 CoT 默认不交易；
- 第三方个人信息、客户商业秘密、代码 IP、员工行为画像都需要单独处理；
- 人员级分析默认关闭，避免默认变成员工监控系统。

### 11.2 产品风险

- 用户看不懂数据价值，只看到复杂表格；
- 数据清洗结果不可信；
- workflow 推荐过多、过噪；
- 训练准备度夸大；
- 数据估值不透明；
- 贡献/交易流程让用户觉得被“拿数据”。

### 11.3 技术风险

- episode 切分错误；
- tool trajectory 不完整；
- 隐式反馈误判；
- 训练数据污染 eval；
- 去重不彻底；
- 脱敏破坏训练价值；
- local-first 计算资源不足；
- 企业侧私有服务复杂度高。

## 14. 最终建议

逐梦应把“数据资产工作台”作为 Agent 产品的核心基础模块，而不是训练模型之前的附属工具。

最小可行路线：

```text
先让数据可见
  -> 再让数据可用于 workflow/skill/eval
  -> 再让数据可生成训练候选
  -> 再让客户可控贡献给逐梦
  -> 最后做数据交易与公共训练池
```

如果这套成立，逐梦的定位会从“AI Agent 工具”升级为：

> AI 工作数据资产化基础设施。

它的护城河不是单点模型能力，也不是单点采集能力，而是把真实 Agent 使用过程沉淀为客户自有资产，并在客户授权下转化为逐梦模型、行业 benchmark 和数据交易网络的能力。

## 15. 路线图与执行边界（避免远期构想干扰当前主线）

> 本节是对本文档的执行边界补充。它不删除前文的中长期构想，但明确：本文不是当前 V0.1/V1 的立即实施清单，而是数据资产化能力的路线图与架构备忘录。

### 15.1 文档定位

本文的正确定位是：

```text
中长期产品路线图 + 架构留缝 + 未来能力备忘录
```

不是：

```text
当前 Sprint 实施计划
当前 V0.1 必做需求
训练平台立刻开工说明
数据交易市场立刻开工说明
```

当前逐梦 Agent 的主线仍然是：

```text
接管现有 Agent 工具
  -> 打通 Codex / Claude Code 等关键入口
  -> 建立 gateway 与采集骨架
  -> 验证个人用户留存
  -> 再进入团队/企业治理面
```

本文所有内容必须服从这个节奏，不得拖慢当前“Agent 接管 + 网关 + 留存验证”的主线。

### 15.2 阶段路线图

| 阶段 | 定位 | 目标 | 可以做 | 明确不做 |
|---|---|---|---|---|
| Phase 0：留缝期 | 现在 | 避免未来返工 | 预埋 owner/tier/data_pool/egress_allowed/contribution_consent_id；登记 dataset_package、出境账本、交易账本等对象 | 不做完整工作台、不做训练流水线、不做交易 |
| Phase 1：数据可见 | 留存验证后、V1 邻近 | 让用户看到自己积累了什么 | 数据健康度、L0-L3 分布、基础 workflow 洞察、单工具/编程域可见性 | 不做 SFT/DPO/RL，不做数据交易，不做人级绩效画像 |
| Phase 2：数据可用 | 团队/企业治理面 | 让数据改善工作流 | workflow 聚类、skill 候选、私有 eval、基础 ROI/token-maxing 看板 | 不进入公共训练池，不做开放市场 |
| Phase 3：数据可贡献 | 授权/治理稳定后 | 让客户可控贡献数据 | dataset_package 预览、贡献授权、出境账本、逐梦共建池入口 | 不做自动贡献，不做默认上传 L2/L3，不交易 L3/L3+ |
| Phase 4：数据可交易 | V2+/V3+，法规与商业验证后 | 让数据具备交易/变现能力 | 估值、逐梦作为收购方、交易账本、收益结算 | 不急做开放数据市场，不做无授权撮合 |
| Phase 5：公共训练增强 | 共建池数据足够且合规稳定后 | 反哺逐梦模型 | 从合规共建池筛选公共训练池，支持模型训练和行业 benchmark | 不绕过授权，不把客户自有池直接训练 |

### 15.3 近期滩头：先单工具 + 编程域 + 数据可见

近期如果要把本文部分能力落地，推荐最窄滩头是：

```text
单工具 / 编程域 / 数据可见
```

优先回答用户能理解的问题：

- 这周我让 Agent 做了什么？
- 哪些任务反复失败？
- 哪些命令、工具、模型消耗最多 token？
- 哪些修改最后被我采纳？
- 哪些任务可以沉淀成 workflow 或 skill 候选？
- 换模型后成功率和成本有没有变化？

近期不应默认展示：

- SFT 候选数量；
- DPO pair 数量；
- RL transition 数量；
- reward model 数据；
- 公共训练池入口；
- 数据交易估值。

这些属于远期留缝或主动开启后的高级能力。

### 15.4 架构收敛：一台精炼机 + 可插拔出料口

本文前文分别描述了 workflow 改进流水线和模型训练数据流水线。执行时不应实现成两套重复系统，而应收敛为：

```text
共享数据精炼核心
  -> workflow 出料口
  -> private eval 出料口
  -> skill 出料口
  -> training candidate 出料口
  -> contribution package 出料口
  -> transaction package 出料口
```

共享数据精炼核心负责：

- ingest；
- 权属/授权校验；
- L0-L3+ 分级；
- PII/secret/IP/商业秘密扫描；
- episode 构建与边界修复；
- 去重；
- 质量评分；
- 自然标注；
- dataset_package manifest 生成。

不同出料口只负责不同用途的视图、导出和授权流程。

### 15.5 dataset_package 作为一等出口对象

后续应把 `dataset_package` 提升为数据资产工作台的一等对象。无论是：

- 私有 eval；
- workflow 包；
- skill 包；
- 训练候选包；
- 贡献包；
- 交易包；

本质上都是：

```text
dataset_package + manifest + consent + tier + destination
```

其中 `destination` 可以是：

```text
customer_owned
private_eval
private_training
zhumeng_co_build
public_training
transaction
```

这样可以避免为每种用途重复设计一套数据模型，也能天然衔接出境账本和交易账本。

### 15.6 Eval-first，而不是 Training-first

V1/V1+ 更推荐从私有 eval 起步，而不是从 SFT/DPO/RL 起步。

原因：

- 私有 eval 风险低；
- 客户价值直接；
- 不需要把数据进入公共训练池；
- 能帮助客户验证“换模型不丢积累”；
- 正好服务数据主权叙事。

训练方法级能力保持留缝：

```text
SFT / DPO / reward model / RL transition / teacher trajectory
```

这些能力只在数据规模、授权链路、删除链路、合规边界和私有 eval 都稳定之后展开。

### 15.7 数据交易降级为远期留缝

数据贡献与交易是重要方向，但近期不应表现为“数据交易所”。

近期只保留：

- dataset_package 预览；
- 客户选择贡献范围；
- 贡献授权；
- 出境账本；
- 逐梦作为收购方的可能性。

远期才考虑：

- 数据估值模型；
- 交易账本；
- 收益结算；
- 第三方撮合；
- 行业数据包市场。

在法规、合同、授权和客户信任机制没有成熟前，不实现开放数据市场。

### 15.8 信任边界

数据资产工作台处理的是最敏感的数据，因此必须继承文档2 的信任缝：

- 默认在本机或租户内运行；
- 原始 L2/L3/L3+ 默认不出域；
- 出境决策和“到底什么离开了”必须可审计；
- 出境账本和贡献授权必须绑定 dataset_package；
- 可开源/可审计的 gateway-lite / shim 继续作为客户信任锚点；
- 云端增强只能处理客户明确允许的数据包。

### 15.9 本地算力降级梯

个人 local-first 不代表所有计算都要实时本机完成。建议降级梯：

| 等级 | 默认 | 说明 |
|---|---|---|
| L0：轻量常驻 | 默认开启 | episode 切分、基础标签、摘要、敏感扫描 |
| L1：本机批处理 | 空闲/夜间运行 | 去重、质量评分、基础 workflow 聚类 |
| L2：延迟深度任务 | 用户手动或每周 | skill 候选、私有 eval 候选、训练候选 |
| L3：可选云协助 | 用户明确开启 | 大模型辅助整理、深度聚类、贡献包估值 |
| L4：企业侧私有服务 | 企业部署 | 大规模聚合、治理看板、组织级 workflow 分析 |

### 15.10 总结

本文的价值是防止未来能力断档，而不是要求当前立刻开工。正确执行方式是：

```text
愿景保留
节奏降档
对象预埋
近期只做数据可见和 eval-first
远期再做训练和交易
```
